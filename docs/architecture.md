# 技术架构详解

## 1. 设计理念

### 为什么选择 Multi-Agent 架构？

| 传统单体 LLM 方案 | 我们的 Multi-Agent 方案 |
|------------------|------------------------|
| 单次调用，信息有限 | 多 Agent 并行，覆盖面广 |
| 观点倾向单一 | Bull/Bear 对抗，观点平衡 |
| 黑盒输出，无法解释 | 每步可追溯，透明可信 |
| 难以扩展新数据源 | 模块化设计，易于扩展 |

### 核心设计原则

1. **分工明确**: 每个 Agent 专注特定任务（检索/辩论/决策）
2. **可溯源**: 每个观点都能追溯到原始文档
3. **对抗生成**: 通过 Bull/Bear 辩论机制，避免单一偏见
4. **渐进决策**: 信息收集 → 观点碰撞 → 综合判断，模拟真实投研流程

## 3. Phase 1: Research Team (研究团队)

### 3.1 架构设计

```
                    用户问题
                        │
                        ▼
              ┌─────────────────┐
              │  意图识别引擎    │  ← 识别问题类型，决定检索策略
              └─────────────────┘
                        │
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ 基本面分析师  │ │ 产业链分析师  │ │ 情绪分析师   │
│              │ │              │ │              │
│ AlphaEngine  │ │ 专家访谈     │ │ Twitter     │
│ (研报/业绩会) │ │ (一手洞察)   │ │ Substack    │
│              │ │              │ │ AceCamp     │
└──────────────┘ └──────────────┘ └──────────────┘
        │               │               │
        └───────────────┼───────────────┘
                        ▼
              ┌─────────────────┐
              │  Research       │  ← 整合多源检索结果
              │  Report         │    标记数据冲突
              └─────────────────┘
```

### 3.2 意图识别

根据用户问题自动识别查询意图，优化检索策略：

| 意图类型 | 触发关键词 | 优先数据源 |
|----------|-----------|-----------|
| `earnings_call` | 业绩会、财报、guidance | AlphaEngine |
| `expert_interview` | 专家、访谈、调研 | 专家访谈 |
| `social_sentiment` | 舆情、看法、观点 | Twitter/Substack |
| `industry_trend` | 行业、趋势、格局 | AceCamp + 专家访谈 |
| `company_outlook` | 未来、展望、预期 | 全部数据源 |

### 3.3 检索策略

采用**两阶段检索**提升召回精度：

1. **Stage 1 - 宽检索**: 使用问题原文 + 公司名变体扩展
2. **Stage 2 - 精准检索**: 基于 Stage 1 结果，提取关键实体再次检索

## 4. Phase 2: Debate Team (辩论团队)

### 4.1 架构设计

```
              Research Report
                    │
                    ▼
    ┌───────────────────────────────┐
    │       Debate Orchestrator     │
    └───────────────────────────────┘
                    │
        ┌───────────┴───────────┐
        ▼                       ▼
┌──────────────┐         ┌──────────────┐
│ 🐂 Bull      │   ⚔️    │ 🐻 Bear      │
│ Analyst      │ ←────→ │ Analyst      │
│              │         │              │
│ 构建看多论点  │         │ 构建看空论点  │
│ 寻找支撑证据  │         │ 寻找风险因素  │
└──────────────┘         └──────────────┘
        │                       │
        └───────────┬───────────┘
                    ▼
           ┌──────────────┐
           │ ⚖️ Moderator │
           │ (研究经理)    │
           │              │
           │ - 评估论点强度 │
           │ - 识别共识点  │
           │ - 标记核心分歧│
           │ - 计算多空比  │
           └──────────────┘
                    │
                    ▼
             Debate Result
```

### 4.2 论点结构

每个论点包含完整的溯源信息：

```python
@dataclass
class Argument:
    point: str          # 论点摘要
    evidence: str       # 支撑证据
    strength: float     # 论点强度 (0-1)
    source: SourceInfo  # 来源溯源
        - source_type   # 数据源类型
        - title         # 文档标题
        - date          # 发布日期
        - file_id       # 知识库文件ID
        - preview       # 原文预览
```

### 4.3 输出结构

```python
@dataclass
class DebateResult:
    bull_arguments: List[Argument]  # 看多论点列表
    bear_arguments: List[Argument]  # 看空论点列表
    moderator_summary: str          # 研究经理总结
    consensus: List[str]            # 多空共识点
    disputes: List[str]             # 核心分歧点
    bull_bear_ratio: float          # 多空比 (0-1)
```

## 5. Phase 3: Decision Agent (决策代理)

### 5.1 架构设计

```
     Research Report + Debate Result
                    │
                    ▼
           ┌──────────────┐
           │  Decision    │
           │  Agent       │
           └──────────────┘
                    │
    ┌───────────────┼───────────────┐
    ▼               ▼               ▼
┌────────┐   ┌────────────┐   ┌────────────┐
│ 多空比  │   │ 论点强度   │   │ 数据冲突   │
│ 加权   │   │ 加权       │   │ 扣分       │
└────────┘   └────────────┘   └────────────┘
    │               │               │
    └───────────────┼───────────────┘
                    ▼
           ┌──────────────┐
           │   Decision   │
           │   Output     │
           └──────────────┘
```

### 5.2 决策输出

```python
@dataclass
class Decision:
    recommendation: Recommendation  # STRONG_BUY/BUY/HOLD/SELL/STRONG_SELL
    confidence: float              # 置信度 (0-100)
    rationale: str                 # 决策理由
    key_risks: List[str]           # 关键风险
    catalysts: List[str]           # 潜在催化剂
    tracking_metrics: List[Metric] # 建议跟踪指标
    time_horizon: str              # 投资时间窗口
    thesis_summary: str            # 完整投资论点
```

### 5.3 置信度计算

置信度受多因素影响：

| 因素 | 影响 |
|------|------|
| 多空比例极端 (>80% 或 <20%) | +15% |
| 论点平均强度高 | +10% |
| 存在数据冲突 | -10% |
| 数据源单一 | -15% |
| 多空共识多 | +5% |

## 6. 数据层详解

### 6.1 知识库 (KB Client)

统一的知识库访问接口：

```python
class KBClient:
    def search(
        self,
        query: str,
        kb_ids: List[str],      # 目标知识库ID列表
        top_k: int = 10,        # 返回数量
        date_filter: str = None # 时间过滤
    ) -> List[Document]
```

### 6.2 数据源映射

| KB ID | 名称 | 内容描述 |
|-------|------|----------|
| `kb_expert` | 专家访谈 | 行业专家一对一访谈纪要 |
| `kb_alpha` | AlphaEngine | 研报、业绩说明会、公司公告 |
| `kb_twitter` | Twitter | KOL 推文、热点讨论 |
| `kb_substack` | Substack | 独立分析师 Newsletter |
| `kb_acecamp` | AceCampTech | 社区共享调研纪要 |

### 6.3 LLM 调用

使用 Claude 作为核心推理引擎：

- **Bull/Bear 论点提取**: 从检索结果中分类提取多空观点
- **论点强度评估**: 评估每个论点的可信度和重要性
- **综合决策生成**: 基于辩论结果生成投资建议

## 7. 扩展性设计

### 7.1 新增数据源

只需实现统一的 `DataSource` 接口：

```python
class DataSource(ABC):
    @abstractmethod
    def search(self, query: str, **kwargs) -> List[Document]:
        pass

    @abstractmethod
    def get_source_type(self) -> str:
        pass
```

### 7.2 新增分析师

在 Research Team 中注册新的分析师角色：

```python
research_team.register_analyst(
    name="技术分析师",
    data_sources=["kb_technical"],
    focus_areas=["K线形态", "技术指标"]
)
```

## 8. 安全设计

- 所有敏感配置（API Key、密码）仅从环境变量读取
- `.env` 文件在 `.gitignore` 中，不提交版本控制
- Demo 模式使用 mock 数据，无需真实凭据
- 知识库访问通过统一网关，支持鉴权和限流
